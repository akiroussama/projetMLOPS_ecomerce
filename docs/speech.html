Slide 2 · Contexte & Donnees · ~1 min
Phase 1 — Le cas d'usage Rakuten
Le cas d'usage : Rakuten France, marketplace e-commerce. Chaque produit doit etre classe dans une des 27 categories, a partir de son titre et de sa description texte.

Dataset de 84 000 produits. Vrai probleme concret : 35% des descriptions sont vides, donc on bosse principalement sur le titre.

La transition ML vers MLOps c'est le tableau a l'ecran. On est passes de notebooks a des scripts modulaires, d'un modele en local a une API, d'une execution a la main a Airflow. C'est ca le coeur du projet.
Slide 3 · Modele Baseline · ~1 min
Phase 1 — TF-IDF + SGDClassifier
Le modele c'est TF-IDF + SGDClassifier. Cinq mille features, unigrams et bigrams, class_weight='balanced'. Ca donne 76% d'accuracy, 74% de F1 macro.

C'est pas le meilleur score — un CamemBERT ferait 85% — mais c'est pas le sujet. L'inference prend moins de 10ms sur CPU, pas de GPU.

Et surtout l'archi est model-agnostic : si demain on met BERT, on change une classe dans train_model.py, tout le reste — API, Airflow, MLflow — ca bouge pas. Le modele c'est un composant interchangeable.

Slide 5 · Docker · ~1 min
Phase 3 — Conteneurisation
Merci Hery. Je prends le relais sur l'intégration logicielle. Mon premier chantier a été de conteneuriser l'API avec une image python:3.10-slim pour minimiser l'empreinte mémoire.

L'optimisation clé ici, c'est que nos modèles Scikit-Learn ne sont pas figés dans l'image Docker. Ils sont montés en volumes et gérés via DVC (Data Version Control) connecté à notre stockage distant. Cela permet de mettre à jour l'intelligence de l'API sans recompiler le conteneur.

J'ai également consolidé le service bootstrap : au premier lancement, il télécharge les données et entraîne le premier modèle automatiquement.

Slide 9 · Airflow · ~1 min
Phase 3 — Orchestration
Merci Johan. L'orchestration c'est la phase 3. On a un DAG — rakuten_weekly_retraining — programme chaque lundi a 2h du matin.

Six taches : telechargement S3, preparation dataset, features TF-IDF, entrainement + log MLflow, verification artefacts, rapport de derive.

C'est pas juste un schema : on l'a declenche sur notre VPS, les 6/6 taches passent au vert. La derniere tache fait un test de Kolmogorov-Smirnov pour detecter si les donnees ont change.

Slide 24b · Demo — GitHub Actions · ~30s
7b. CI/CD — Ce qu'on détecte
Concrètement, ce genre de pipeline attrape trois catégories d'erreurs. Le code mort — imports inutilisés, variables non définies — que le linter détecte avant même l'exécution. Les régressions — une fonction qu'on modifie et qui casse quelque chose ailleurs — que les tests capturent automatiquement. Et les problèmes de build — une image Docker qui ne compile plus à cause d'une dépendance manquante.

Le principe : plus on détecte tôt, moins ça coûte. Une erreur attrapée au commit, c'est cinq minutes de fix. La même erreur en production, c'est une coupure de service.

Conclusion — Ce qu'on a construit & les limites
Pour conclure. Cinq phases couvertes — les trois obligatoires et les deux optionnelles. Mais au-delà de la checklist, ce qu'on retient c'est ce qu'on a appris en le construisant : gérer huit conteneurs qui communiquent, les permissions, les volumes partagés, les incompatibilités de librairies. Q&A · Pourquoi pas BERT ?
Pourquoi un modèle linéaire et pas BERT ?
TF-IDF + SGD : 76% de précision, 5ms de réponse.
BERT sur CPU : ~200ms — 40 fois plus lent, pour +5 à 10% de gain.
Sur une marketplace avec des millions de requêtes, la latence compte.

Et notre architecture est model-agnostic : on change une classe Python, le pipeline reste identique. Si on a un GPU demain, on switche.
Q&A · Drift en production
Votre détection de drift est-elle réelle ?
Non, et on l'a dit en conclusion. On compare train vs validation du même dataset — évidemment pas de drift.

En production réelle : on loggerait les inputs utilisateurs depuis l'API, et on comparerait leur distribution avec les données d'entraînement via PSI ou KL-divergence. C'est le prochain pas.
Q&A · Pourquoi pas Kubernetes ?
Pourquoi Docker Compose et pas Kubernetes ?
8 services, 1 serveur — Kubernetes est une surcharge injustifiée ici.
Docker Compose suffit, et ça reste lisible et maintenable.

Si on scale : les images Docker sont identiques, on change juste la cible de déploiement. L'architecture est cloud-agnostic par construction.
Q&A · 36% de confiance
36% de confiance, c'est pas faible ?
La baseline aléatoire sur 27 classes, c'est 3.7%.
On est 10 fois au-dessus du hasard sur cette prédiction.

Un modèle binaire peut donner 95% de confiance. Avec 27 classes et un modèle linéaire sur texte court, c'est attendu. Ce qui compte, c'est l'accuracy globale : 76%.
Q&A · C'est quoi le F1 macro ?
C'est quoi le F1 macro à 73.9% ?
Le F1 macro, c'est la moyenne du F1 sur chaque classe, sans pondération.
Ça veut dire que les petites catégories (peu de produits) comptent autant que les grandes.

73.9% signifie que le modèle performe bien même sur les catégories rares — pas seulement sur les plus fréquentes. C'est pour ça qu'on a utilisé class_weight = balanced.
Q&A · Sécurité API
Comment sécurisez-vous l'API ?
Bearer token sur /predict. Sans token : 401, on l'a vu dans Grafana.
/health et /metrics sont publics par design — Prometheus en a besoin.

Le token est en variable d'environnement, jamais dans le code. Le VPS a un reverse proxy pour le HTTPS.
Q&A · Réentraînement auto
Le modèle se réentraîne-t-il automatiquement ?
Le DAG Airflow peut tourner sur schedule — le pipeline complet prend 60 secondes.

Ce qu'on n'a pas encore : le hot-swap sans restart de l'API, et le rollback automatique via MLflow Model Registry avec des stages Staging / Production. C'est la prochaine étape.
Q&A · Pourquoi SQLite pour MLflow ?
Pourquoi SQLite et pas PostgreSQL pour MLflow ?
SQLite suffit pour un contexte mono-serveur avec une équipe de 3 personnes.
Pas de concurrence d'écriture, pas besoin d'un service géré.

En production : on passe sur PostgreSQL ou un managed DB — c'est un paramètre de configuration dans MLflow, pas un refactoring.
Q&A · Variables d'environnement
Comment vous gérez les variables d'environnement ?
Trois niveaux.

Développement : un fichier .env local, jamais commité — il est dans le .gitignore. C'est là qu'on met le token API, les mots de passe Airflow, les credentials.

Docker Compose : les variables sont injectées via la directive env_file ou environment dans le docker-compose.yml. Chaque service reçoit uniquement ce dont il a besoin — le principe du moindre privilège.

CI/CD : les secrets sont stockés dans GitHub Actions Secrets, jamais en clair dans le code. Le pipeline les injecte au moment du build.
Q&A · RGPD
Votre API est-elle RGPD compliant ?
Dans notre contexte : on ne stocke aucune donnée personnelle. L'API reçoit un texte produit, retourne une catégorie, et ne conserve rien. Pas de base utilisateurs, pas de logs nominatifs.

Ce qu'on logge : des métriques agrégées — nombre de requêtes, latence, codes HTTP. Aucune donnée permettant d'identifier un utilisateur.

En production Rakuten, le vrai sujet RGPD serait sur les données d'entraînement : s'assurer que les descriptions produits ne contiennent pas de données personnelles avant d'entraîner le modèle.
Q&A · Qu'est-ce que vous changeriez ?
Qu'est-ce que vous feriez différemment en prod ?
Trois choses :

1. Drift sur vrais inputs — logger les requêtes API et comparer avec les données d'entraînement.
2. Rollback automatique — MLflow Model Registry avec stages Staging / Production.
3. Monitoring des données, pas seulement des métriques HTTP — détecter quand les textes entrants changent de nature.
